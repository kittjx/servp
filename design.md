医院后勤服务管理平台详细设计文档

一、项目概述
1.1 项目背景
当前医院后勤报修依赖电话、微信等非结构化方式，缺乏统一记录、追踪与统计手段，导致响应不及时、责任不清、数据无法沉淀。

1.2 项目目标
构建集“报修、调度、处理、验收、统计”于一体的闭环服务管理平台，实现：
- 一键报修
- 自动/人工派单
- 全流程可追溯
- 多维度数据统计与展示

1.3 覆盖科室
- 信息科
- 设备科
- 总务科
- 动力科

二、系统架构设计
2.1 总体架构
采用前后端分离，支持高并发、高可用部署。

+-------------------+       +-------------------+       +-------------------+
| 微信小程序端       |  <--> | FastAPI后端服务     | <-->  | PostgreSQL + Redis |
| (医护/工程师端)    |       | (业务逻辑层)        |       |                    |
+-------------------+       +-------------------+       +-------------------+
                                    ^
                                    |
                            +-------------------+
                            | PC管理后台.         |
                            | (Vue 3 + Element) |
                            +-------------------+

2.2 技术选型
- 前端：Uniapp + Vue3 + Pinia（状态管理）+ Vant Weapp（组件）  
- 后端：FastAPI + SQLModel + JWT + Celery  
- 数据库：PostgreSQL + Redis（缓存/消息队列）  
- 部署：Docker + Docker Compose + Nginx  

三、角色与权限设计（RBAC）
| 角色 | 权限说明 |
|------|----------|
| 报单人 | 扫码报修、进度查询、验收、评价 |
| 处理工程师 | 接收任务、处理、转派、填写记录 |
| 科室调度员 | 查看本科室工单、改派、监控工作量 |
| 院领导 | 查看驾驶舱、统计报表、满意度分析 |
| 系统管理员 | 组织架构、权限配置、数据字典维护 |

四、核心业务流程设计
- 报修流程：填写问题描述（图文/语音） → 提交 → 系统自动/人工调度  
- 处理流程：工程师接收 → 到场处理 → 上传处理记录 → 完工 → 通知报单人验收  
- 验收流程：报单人确认完成 → 满意度评价 → 流程闭环  

五、功能模块设计
5.1 微信小程序（医护/报单端）
- 快捷填单：拍照、录音、文字描述
- 进度查询：查看工单状态
- 验收评价：完工确认与评分

5.2 微信小程序（工程师端）
- 任务工作台：待办/处理中任务列表
- 工单处理：上传照片、填写配件、申请转派
- 消息通知：任务、转派、超时提醒

5.3 PC管理后台
- 工单管理：列表、检索、导出Excel  
- 调度配置：自动派单规则、手动改派  
- 数据看板：服务驾驶舱、部门效率、满意度趋势  
- 系统设置：科室树、用户管理、角色权限、数据字典  

六、关键业务规则
- 自动派单：规则/条件匹配  
- 差评/超时预警：定时任务 + 消息推送  

七、非功能性设计
- 易用性：小程序报单 ≤3步，界面简洁  
- 稳定性：接口限流、熔断，数据库读写分离，消息队列防丢单  
- 扩展性：预留API接口，支持HIS、资产系统、第三方维保集成

十、前端页面设计
10.1 微信小程序页面
10.1.1 报修首页
功能：报修入口、历史工单
布局：
顶部：用户信息、消息中心
中部：扫码报修大按钮
下方：最近工单列表（3条）
底部导航：首页、工单、我的

10.1.2 报修页
功能：报修创建工单
字段：
问题描述（文本输入，必填）
问题分类（下拉选择）
优先级（单选）
拍照上传（最多9张）
语音录入（按钮）
操作：提交按钮（置灰直到问题描述填写）

10.1.3 工单详情页
顶部：工单状态时间线
信息区：问题描述、附件
处理记录：时间轴展示处理过程
底部操作：根据状态显示不同按钮
待处理：催单
待验收：确认完成、不满意

10.2 PC管理后台页面
10.2.1 工单管理列表
筛选区：科室、状态、时间、处理人、关键词
列表区：工单号、问题、状态、优先级、处理人、时间
操作区：查看详情、改派、导出
统计卡片：今日工单、处理中、已完成、超时

10.2.2 工单详情页
基本信息：工单号、报修人、报修时间、设备信息
处理流程：时间轴展示完整流程
处理记录：工程师操作记录
评价信息：满意度评分、评价内容
操作区：改派、关闭工单

10.2.3 调度配置页
规则列表：问题类型 → 处理科室/工程师
新增规则：选择问题类型、选择处理部门、选择处理人（可选）
优先级规则：根据设备类型自动设置优先级

10.2.4 数据看板
驾驶舱：ECharts图表展示
今日工单趋势（折线图）
工单状态分布（饼图）
科室效率对比（柱状图）
满意度趋势（折线图）
报表导出：支持PDF/Excel导出

十一、消息通知
- 微信小程序模板消息 + 短信通知  
- 支持新工单、超时、差评、待验收、转派提醒  

十二、SLA与优先级
| 优先级 | 响应时间 | 处理时间 | 场景 |
|--------|----------|----------|------|
| 特急 | 5分钟 | 30分钟 | 生命支持设备、停电、网络中断 |
| 紧急 | 15分钟 | 2小时 | 重要设备故障、影响诊疗 |
| 普通 | 30分钟 | 4小时 | 一般办公设备、非紧急问题 |

十三、部署架构
- Docker Compose + Nginx  
- Redis缓存、PostgreSQL主从  

十四、安全设计
- 数据加密存储（手机号、密码）  
- HTTPS传输、JWT认证、RBAC接口权限  
- 防攻击：接口限流、SQL注入、XSS、CSRF防护  

十五、开发计划与风险
- 开发周期：需求确认1周 → 数据库1周 → 后端4周 → 前端4周 → 联调2周 → UAT 1周 → 上线1周  
- 风险：高并发、数据丢失、接口性能、用户抵触  
- 应对：限流、消息队列、缓存优化、培训激励
